## 1. 건축선 핵심 로직 (buildingLine.ts)

### 1-1. 도로폭 실측 [중요도: 높음]

현재 `roadWidth: 4`로 하드코딩. `getRoadSetback()`이 도로폭에 따라 다른 값을 리턴하도록 설계되어 있지만 입력값이 항상 4라서 무의미.

**해야 할 것:**
- 도로 필지(jimok='도') 폴리곤이 있으면, 대지 경계선과 맞닿은 변의 수직 방향으로 도로 폴리곤의 너비를 계산
- 도로 필지가 없는 경우(빈 공간 휴리스틱), 대지 변의 중점에서 반대편 필지까지의 거리로 도로폭 추정
- 결과를 `analyzeBuildingLine()`에 전달

**관련 코드:**
- `buildingLine.ts` → `findRoadAdjacentEdges()` 156번째 줄 `roadWidth: number = 4`
- `setbackTable.ts` → `getRoadSetback()` — 4m 미만이면 중심선에서 2m 후퇴

### 1-2. 가각전제(코너컷) 구현 [중요도: 중간]

`CORNER_CUT_RULES`가 `setbackTable.ts`에 정의되어 있지만 호출하는 곳 없음.

**해야 할 것:**
- 두 도로가 만나는 모서리(꼭짓점) 탐지
- 두 도로 폭의 합에 따른 가각전제 거리 계산 (2m / 3m / 4m)
- 건축선 폴리곤에서 해당 꼭짓점을 대각선으로 잘라내기
- `calculateBuildingLine()` 결과에 반영

**관련 코드:**
- `setbackTable.ts` → `CORNER_CUT_RULES` (179~183줄)

### 1-3. 막다른 도로 판별 [중요도: 낮음]

`getRoadSetback()`에 `isDeadEnd`, `deadEndLength` 파라미터 있지만 항상 기본값(false, 0)으로 호출.

**해야 할 것:**
- 도로 네트워크를 분석하여 막다른 도로 여부 판별
- V-World 도로 레이어 활용 가능성 검토
- 난이도 높아 졸업 프로젝트 범위에서 제외 고려

### 1-4. 다중 블록 건축선 [중요도: 중간]

현재 `selectedBlocks[0]` 첫 번째 블록만 분석.

**해야 할 것:**
- 여러 블록 선택 시 `turf.union()`으로 합필
- 합필된 폴리곤으로 건축선 분석 실행
- `useBuildingLine.ts` → `calculateBuildingLine()` 92~93줄 수정

### 1-5. 건물 용도 선택 연결 [중요도: 높음]

`buildingUse`가 항상 `'기타'`로 고정. 용도에 따라 이격거리가 달라짐.

**해야 할 것:**
- 사이드바에 건물 용도 드롭다운 추가 (공장/판매시설/숙박시설/공동주택/단독주택/근린생활/업무/기타)
- 선택 값을 store에 저장
- `analyzeBuildingLine()` 호출 시 전달
- 프로젝트 저장/불러오기에도 반영

## 2. 규정 검토 연동 (validation.py ↔ 프론트)

### 2-1. 프론트 건축선 → 백엔드 검토 연동 [중요도: 높음]

프론트에서 계산한 건축선 결과(도로변/인접대지변 구분, 이격거리)를 백엔드에 전달하는 흐름 없음. 백엔드는 단순 대지경계선 기준 이격거리만 계산.

**해야 할 것:**
- `/api/validate` 엔드포인트에 건축선 분석 결과 포함하여 전송
- 백엔드에서 변별 이격거리 기준으로 검토
- 또는 프론트에서 건축선 기반 검토를 직접 수행 (건물 폴리곤과 건축선 폴리곤 비교)

**관련 코드:**
- `backend/services/validation.py` → `validate_placement()`
- `frontend/lib/api.ts` → `validatePlacement()`

### 2-2. 용적률(FAR) 검토 [중요도: 중간]

`ValidationConfig`에 `far_limit` 필드가 있지만 `None`으로 미구현.

**해야 할 것:**
- 건물 연면적 = 바닥면적 × 층수
- 용적률 = (연면적 / 대지면적) × 100
- 용도지역별 용적률 한도 데이터 추가

### 2-3. 일조권 사선제한 [중요도: 낮음]

설계서에 언급되어 있지만 코드에 없음.

**해야 할 것:**
- 북측 인접대지 경계선 판별
- 높이에 따른 사선 제한 각도 계산
- 3D 시각화 (사선 제한 면 표시)
- 난이도 매우 높아 범위 축소 또는 제외 고려

## 3. 프론트엔드 UI

### 3-1. 건축선 분석 결과 사이드바 표시 [중요도: 높음]

현재 콘솔에만 로그. 사용자에게 보이지 않음.

**해야 할 것:**
- 사이드바 validate 탭에 건축선 결과 패널 추가
- 표시 내용: 용도지역, 도로변 개수, 인접대지변 개수, 각 변의 이격거리
- 건폐율/이격거리 검토 결과와 함께 표시

**관련 코드:**
- `frontend/components/Sidebar.tsx`
- `useBuildingLine` 훅의 `buildingLineResult`, `currentZoneType` state

### 3-2. 용도지역 화면 표시 [중요도: 중간]

`fetchZoneType()`으로 조회한 값이 `currentZoneType` state에만 저장.

**해야 할 것:**
- CesiumViewer 상단 또는 사이드바에 현재 용도지역명 표시
- 색상 코드로 용도지역 구분 (주거=초록, 상업=빨강, 공업=보라 등)

### 3-3. 건축선 토글 버튼 [중요도: 높음]

`toggleBuildingLine()`이 있는데 UI에서 호출하는 경로 확인 필요.

**해야 할 것:**
- 사이드바 또는 CesiumViewer 컨트롤 패널에 건축선 ON/OFF 버튼
- 블록 선택 후 버튼 활성화
- 계산 중 로딩 표시

### 3-4. 에러 핸들링 개선 [중요도: 중간]

건축선 계산 실패 시 `console.error`만 찍고 끝.

**해야 할 것:**
- `ErrorBanner` 컴포넌트로 에러 표시 연결
- WFS API 실패, 용도지역 조회 실패 등 개별 에러 메시지
- 재시도 버튼

## 오늘 완료한 작업 (2026-03-21)

### buildingLine.ts 수정 (293줄 → 479줄)
- `edgesShareBoundary()`: 중점 1개 → 25%/50%/75% 3점 샘플링, tolerance 0.5m → 1.5m
- `findRoadAdjacentEdges()`: 3단계 판별 (명시적 도로 → 비도로 필지 → 빈 공간=도로 휴리스틱)
- `isSameParcel()`: PNU 기반 자기 필지 제외
- `offsetEdgeInward()`: 변을 내부 방향으로 평행이동 (법선벡터 + 위도보정)
- `lineIntersection()`: 오프셋 라인 교점 계산
- `calculateBuildingLine()`: 변별 개별 이격거리 오프셋 → 교점 폴리곤 → intersect 클리핑

### useBuildingLine.ts 수정 (318줄 → 354줄)
- `adjacentEdgeEntitiesRef` 분리 (도로변/인접대지변 ref 혼용 버그 수정)
- 건축선 시각화: 빨간 대시선 + 반투명 건축가능영역 면
- 도로변: 주황 8px, 인접대지변: 노란 4px
- 복원(restore)도 동일 패턴 적용
