## 이슈 1: turf.buffer 음수 offset 불안정

**파일**: `buildingLine.ts` → `calculateBuildingLine()`

**증상**: 건축선 계산 시 `turf.buffer(polygon, -distance)`로 내부 offset을 적용하면, 복잡한 형태의 필지에서 결과가 null이 되거나 비정상적인 폴리곤이 생성됨.

**원인**: Turf.js의 `buffer()`는 공식적으로 음수 값(내부 축소)을 지원하지 않음. 내부적으로 JSTS 라이브러리를 사용하는데, 음수 buffer 시 자기교차(self-intersection)가 발생하면 빈 결과를 반환함.

**현재 코드**:
```typescript
// buildingLine.ts:178
const buffered = turf.buffer(cadastralPolygon, -uniqueSetbacks[0], { units: 'meters' })
```

**대안**:
1. 백엔드 Shapely의 `polygon.buffer(-distance)`를 사용 (더 안정적)
2. 클라이언트에서 각 edge를 개별적으로 내부 평행이동하는 알고리즘 직접 구현
3. 현재는 단순한 형태의 필지에서만 정상 동작 — 복잡한 형태에서 null 처리 필요

**상태**: ⚠️ 미해결 (단순 필지에서는 동작, 복잡한 형태에서 불안정)

## 이슈 2: 변별 개별 이격거리 미적용

**파일**: `buildingLine.ts` → `calculateBuildingLine()`

**증상**: 도로 접촉 변과 인접 대지 변의 이격거리가 다른데, 실제 계산에서는 최소값으로 일괄 적용됨. 예를 들어 도로 쪽은 1m, 인접 대지 쪽은 0.5m이어야 하는데 전부 0.5m로 계산됨.

**원인**: 각 edge별 개별 offset 계산이 구현되지 않음. turf.buffer는 전체 폴리곤에 균일한 offset만 적용 가능.

**현재 코드**:
```typescript
// buildingLine.ts:182-183
// 이격거리가 다른 경우: 최소 이격거리로 일괄 적용 (간소화)
// TODO: 각 edge별 개별 offset은 복잡하므로 백엔드 Shapely 권장
const minSetback = Math.min(...setbacks.filter(s => s > 0), DEFAULT_SETBACKS.fromAdjacentLot)
```

**대안**:
1. 백엔드에서 Shapely로 각 edge별 개별 buffer 적용 후 intersection으로 건축 가능 영역 산출
2. 또는 클라이언트에서 각 edge를 법선 방향으로 개별 이동 후 교점으로 새 폴리곤 구성

**상태**: ⚠️ 미해결 (TODO 주석으로 남겨둔 상태)

## 이슈 3: 도로 인접 판별 정확도 — midpoint 방식의 한계

**파일**: `buildingLine.ts` → `edgesShareBoundary()`

**증상**: 대지 경계의 특정 변이 도로와 접해 있는데 도로로 인식되지 않거나, 반대로 도로가 아닌데 도로로 인식되는 경우가 있음.

**원인**: 현재 edge의 중점(midpoint)과 도로 필지 경계선 간의 거리만으로 판단함 (tolerance: 0.5m). 변의 일부만 도로와 접하거나 변이 긴 경우 중점이 도로와 멀어서 놓칠 수 있음.

**현재 코드**:
```typescript
// buildingLine.ts:97-104
const midpoint = turf.midpoint(
  turf.point([edge.geometry.coordinates[0][0], ...]),
  turf.point([edge.geometry.coordinates[1][0], ...])
)
const distance = turf.pointToLineDistance(midpoint, polygonLine, { units: 'meters' })
return distance <= tolerance  // 0.5m
```

**대안**:
1. midpoint만이 아니라 edge의 시작점/끝점/1/4지점/3/4지점 등 여러 포인트에서 검사
2. `turf.lineOverlap()` 또는 `turf.booleanOverlap()`으로 실제 겹침 구간 확인
3. tolerance를 1~2m로 늘려서 지적도 좌표 오차를 수용 (단, 좁은 필지에서 오탐 위험)

**상태**: ⚠️ 미해결 (대부분 필지에서 동작하나 엣지 케이스 존재)

## 이슈 5: 도로폭 판별 미구현 (하드코딩 4m)

**파일**: `useBuildingLine.ts`, `buildingLine.ts`

**증상**: 모든 도로를 4m 이상으로 가정하여 건축선 후퇴가 적용되지 않음. 4m 미만 도로에서는 도로중심선에서 2m 후퇴해야 하는데 이 로직이 실행되지 않음.

**원인**: 도로폭을 조회할 수 있는 API 연동이 없음. `useBuildingLine.ts`에서 `roadWidth: 4`로 하드코딩.

**현재 코드**:
```typescript
// useBuildingLine.ts:121-123
const result = await analyzeBuildingLine(
  cadastralPolygon, nearbyParcels, zoneType,
  '기타', 4  // ← roadWidth 하드코딩
)
```

**대안**:
1. V-World 도로중심선 레이어(`lt_l_n3a0020000`)의 도로폭 속성 활용
2. 사용자에게 도로폭 직접 입력받는 UI 추가 (폴백)
3. 현재는 대부분 4m 이상 도로이므로 영향 제한적

**상태**: ⚠️ 미구현 (계획 문서에 LOW 우선순위로 기재)

## 이슈 7: 좌표 변환 근사치 사용

**파일**: `buildingLine.ts` → `expandBbox()`

**증상**: 주변 필지 조회를 위해 bbox를 50m 확장할 때 `1/111320` 근사치를 사용하여 위도에 따라 오차 발생.

**현재 코드**:
```typescript
// buildingLine.ts:285
const bufferDeg = bufferMeters / 111320  // 대략적인 미터→도 변환
```

**문제점**: 위도 37도(서울) 기준 경도 1도 ≈ 88.8km이므로 경도 방향 오차가 약 20% 발생. 50m 확장 시 경도 방향으로는 실제 약 40m만 확장됨.

**대안**:
```typescript
const bufferLat = bufferMeters / 111320
const bufferLon = bufferMeters / (111320 * Math.cos(centerLat * Math.PI / 180))
```

**상태**: ⚠️ 미해결 (실용적으로는 50m 버퍼가 충분히 커서 큰 문제 없음)

## 이슈 8: 프론트엔드-백엔드 이격거리 로직 이중화

**파일**: `frontend/lib/setbackTable.ts` vs `backend/services/validation.py`

**증상**: 이격거리 기준값이 프론트엔드와 백엔드에 각각 별도로 정의되어 있고, 값이 일치하지 않는 부분이 있음.

**불일치 예시**:

| 용도지역 | 프론트엔드 (setbackTable.ts) | 백엔드 (validation.py) |
|---------|---------------------------|----------------------|
| 제1종전용주거 | 건축선 1m + 인접대지 0.5m | setback_required: 2.0m |
| 일반상업지역 | 건축선 0m + 인접대지 0m | setback_required: 0.0m |

**원인**: 프론트엔드는 건축법 시행령 별표 2 기준(건축선/인접대지 구분), 백엔드는 단순화된 단일 이격거리값 사용. 두 시스템의 관점이 다름.

**대안**:
1. 백엔드 validation.py를 건축선/인접대지 구분 방식으로 확장
2. 또는 프론트엔드 건축선 계산 결과를 백엔드 validate-placement에 전달하는 방식으로 통합
3. 최소한 기준값 출처를 한 곳에서 관리 (공통 JSON 설정 파일 등)

**상태**: ⚠️ 구조적 이슈 (당장 기능에는 문제없으나 유지보수 시 혼란 가능)

## 이슈 9: buildableArea가 buildingLine과 동일하게 반환됨

**파일**: `buildingLine.ts` → `analyzeBuildingLine()`

**증상**: 건축 가능 영역(`buildableArea`)과 건축선(`buildingLine`)이 동일한 값을 반환함. 실제로는 건축선은 "넘으면 안 되는 선"이고, 건축 가능 영역은 건축선 + 건폐율 + 기타 제약을 모두 반영한 영역이어야 함.

**현재 코드**:
```typescript
// buildingLine.ts:231
buildableArea: buildingLine,  // 현재는 건축선 = 건축 가능 영역
```

**대안**: 건축 가능 영역 계산을 별도로 구현하거나, 최소한 인터페이스상 구분을 명확히 하여 향후 확장 시 혼동 방지.

**상태**: ⚠️ 미구현 (설계상 구분은 되어 있으나 실제 로직 미분리)

## 이슈 10: WFS 응답의 도로 필지 식별 방식

**파일**: `buildingLine.ts` → `findRoadAdjacentEdges()`

**증상**: `nearbyParcels.filter(p => p.properties?.isRoad === true)`로 도로 필지를 필터링하는데, 실제 V-World WFS 응답에는 `isRoad` 속성이 없음. 지적도 WFS의 `jimok` (지목) 필드에서 "도" 값으로 판별해야 함.

**현재 코드**:
```typescript
// buildingLine.ts:122
const roadParcels = nearbyParcels.filter(p => p.properties?.isRoad === true)
```

**예상 수정**:
```typescript
const roadParcels = nearbyParcels.filter(p =>
  p.properties?.jimok === '도' || p.properties?.jibun?.includes('도로')
)
```

**상태**: 🔴 버그 (WFS 응답 구조와 불일치 — 도로 필지가 0개로 필터링될 가능성 높음)
