## 2. 요구사항 정의

### 2.1 서비스 구성 및 개요

본 시스템은 건축 설계 초기 단계에서 CAD 도면(DXF)을 기반으로 3D 건물 매스를 자동 생성하고, 실제 지도 위에서 건물 배치 및 건축 규정 적합 여부를 검토할 수 있는 웹 기반 서비스이다. 사용자는 웹 브라우저를 통해 접속하며, 건축 전문가가 아닌 일반 사용자도 직관적으로 건축 규정 검토 결과를 확인할 수 있도록 설계한다.

서비스는 아래 4개 주요 구성요소로 이루어진다.

1) 클라이언트 (Next.js + CesiumJS)
   - 사용자 인터페이스 및 3D 지도 시각화를 담당한다
   - CesiumJS 기반 3D 뷰어에서 건물 배치, 이동, 회전, 높이 조절 등 실시간 인터랙션을 처리한다
   - Zustand 상태 관리를 통해 건물 조작 데이터를 클라이언트에서 관리하고, 필요 시 서버에 전송한다
   - 일조 시뮬레이션(CesiumJS 내장 엔진), OSM Buildings(주변 건물 3D), 지적도 오버레이 등을 제공한다

2) 서비스 서버 (FastAPI)
   - DXF 파일 업로드 및 풋프린트 파싱, 3D GLB 모델 생성, 건축 규정 검토 등 핵심 비즈니스 로직을 수행한다
   - RESTful API를 통해 클라이언트와 통신하며, Pydantic 모델로 데이터 검증을 수행한다
   - CAD 처리(ezdxf), 기하 연산(Shapely), 좌표 변환(pyproj), 3D 모델 생성(trimesh) 등의 파이프라인을 제공한다

3) AI 학습서버 (설계 예정)
   - DXF 도면에서 건물 외곽선을 자동 추출하는 AI 모델을 학습하고 서빙한다
   - 학습 모델 후보: XGBoost, MLP, Random Forest (확정 예정)
   - 서비스 서버와 분리 운영되며, REST API를 통해 추론 요청을 처리한다

   > ⚠️ AI 모델 종류 및 학습 방식 미확정. 확정 후 상세 설계 추가 예정.

4) 데이터베이스 (PostgreSQL + PostGIS, 설계 예정)
   - 사용자 정보, 프로젝트 데이터, 건물 배치 정보, 분석 결과 등을 영구 저장한다
   - PostGIS 확장을 통해 공간 데이터(좌표, 폴리곤)를 효율적으로 저장·조회한다

   > ⚠️ DB 스키마는 서버 구축 및 AI 파트 확정 후 설계 예정.

서비스 절차:
사용자가 DXF 도면을 업로드하면, 서비스 서버가 도면을 파싱하여 건물 풋프린트를 추출한다. 추출된 풋프린트를 기반으로 3D GLB 매스 모델을 생성하고, 클라이언트의 CesiumJS 뷰어에서 렌더링한다. 사용자는 3D 환경에서 건물 위치, 회전, 높이를 조정한 뒤 건축 규정 검토(건폐율, 이격거리, 높이제한)를 실행한다. 검토 결과는 항목별 적합/부적합으로 표시되며, 일조 시뮬레이션을 통해 그림자 영향도 확인할 수 있다. 최종 결과는 프로젝트로 저장하거나 PDF/이미지로 내보낼 수 있다.

(※ 서비스 시나리오 구성도는 PPT로 작성 후 캡처하여 삽입. 반드시 클라이언트, 서비스 서버, AI학습서버, DB를 분리하여 표현할 것.)

### 2.2 데이터분석의 요구사항

#### ■ 서비스 운영 데이터

| No | 데이터명 | 형식 | 출처 | 용도 | 설명 |
|----|----------|------|------|------|------|
| 1 | DXF 도면 파일 | .dxf (Drawing Exchange Format) | 사용자 업로드 | 건물 풋프린트 추출, 3D 매스 생성 | AutoCAD 호환 2D 도면 파일. ezdxf 라이브러리로 파싱하여 폴리라인/폴리곤 엔티티에서 건물 외곽선을 추출한다 |

| 4 | 주변 건물 데이터 | 3D Tiles | Cesium Ion (OSM Buildings) | 주변 건물 3D 표시, 일조 분석 참고 | OpenStreetMap 기반 전세계 건물 3D 타일셋. 건물 높이와 풋프린트 정보를 포함한다 |
| 5 | 위성/항공 이미지 | WMTS Tile | Cesium Ion (Bing Maps Aerial) | 지도 배경 이미지 | 위성사진 기반 지도 타일. 사용자가 실제 지형을 시각적으로 확인하는 데 사용한다 |
| 6 | 건축 규정 기준 데이터 | JSON (내부 설정) | 건축법 시행령, 국토계획법 | 건폐율/이격거리/높이제한 검토 | 용도지역별 건축 규정 기준값. 현재 8개 용도지역(제1종전용주거~준공업) 기본값을 시스템에 내장하며, 관리자가 추가/수정할 수 있다 |

### 2.4 AI서비스 유스케이스 모델

#### ■ 액터(Actor) 정의

| 액터 | 설명 |
|------|------|
| 일반 사용자 (User) | 웹 브라우저를 통해 건물 배치 설계 및 규정 검토 서비스를 이용하는 건축 설계자, 기획 담당자, 일반 사용자 |
| 관리자 (Admin) | 시스템 운영 및 사용자/데이터를 관리하는 운영 담당자. 관리자 전용 대시보드를 통해 접근한다 |
| 서비스 서버 (FastAPI) | DXF 파싱, 3D 모델 생성, 건축 규정 검토 등 핵심 비즈니스 로직을 수행하는 백엔드 서버 |
| AI 학습서버 | DXF 레이어 분류 AI 모델의 학습, 추론, 배포를 담당하는 독립 서버 |
| 데이터베이스 (PostgreSQL) | 사용자, 프로젝트, 건물, 분석 결과 등의 영구 데이터를 저장·관리하는 DB 서버 |
| 외부 데이터 서비스 | Cesium Ion(지형/건물 3D), 국가공간정보포털(지적도 WFS), Geocoding API 등 외부 데이터 제공자 |

#### ■ 사용자용 유스케이스 다이어그램

(※ 그림 2.2 사용자용 유스케이스 다이어그램 — PPT 또는 UML 툴로 작성 후 캡처하여 삽입)

사용자용 유스케이스는 아래와 같이 7개 주요 기능 그룹으로 구성된다.

**UC-01. 사용자 인증**
사용자는 이메일 기반 회원가입, 로그인, 로그아웃을 수행한다. 로그인 시 JWT 토큰을 발급받으며, 이후 모든 API 요청에 인증 토큰을 포함한다. 서비스 서버는 토큰의 유효성을 검증하고, 만료 시 재로그인을 요구한다.

**UC-02. 프로젝트 관리**
사용자는 건축 검토 프로젝트를 생성·저장·불러오기할 수 있다. 프로젝트에는 대지 정보, 건물 배치 데이터, 규정 검토 결과 등이 포함된다. 프로젝트 데이터는 데이터베이스에 영구 저장되며, 사용자는 이전에 작업한 프로젝트를 목록에서 선택하여 이어서 작업할 수 있다.

**UC-03. CAD 도면 관리**
사용자는 DXF 형식의 CAD 도면을 업로드한다. 서비스 서버는 ezdxf 라이브러리를 사용하여 도면을 파싱하고 건물 풋프린트(외곽선 좌표)를 추출한다. AI 학습서버가 연동된 경우, DXF 레이어 분류 AI 모델을 통해 외벽 레이어를 자동 식별하여 추출 정확도를 높인다. 사용자는 업로드된 파일을 삭제하거나 2D 미리보기를 확인할 수 있다.

**UC-04. 3D 지도 조작 및 부지 선택**
사용자는 CesiumJS 기반 3D 지도에서 마우스 조작으로 시점을 이동·확대·축소·회전한다. 지도 위에서 클릭하여 분석 대상 부지를 선택하면, 외부 데이터 서비스(국가공간정보포털 WFS)와 연동하여 필지 경계와 지적 정보를 오버레이한다. 주소 또는 좌표 입력을 통한 부지 검색 기능도 제공한다.

**UC-05. 건물 매스 관리**
사용자는 DXF에서 추출한 풋프린트를 기반으로 3D 건물 매스를 생성한다. 서비스 서버는 trimesh 라이브러리로 GLB 형식 3D 모델을 생성하고, CesiumJS 뷰어에서 렌더링한다. 사용자는 3D 환경에서 건물의 위치를 드래그로 이동하고, 회전(Z축)을 조정하며, 슬라이더로 높이를 변경할 수 있다. 모든 조작 데이터는 Zustand 상태 관리를 통해 클라이언트에서 실시간 반영된다.

**UC-06. 환경 확인 및 시뮬레이션**
사용자는 OSM Buildings 데이터를 통해 주변 기존 건물을 3D로 확인하고, Cesium World Terrain을 통해 지형 고저차를 파악한다. 날짜와 시간을 설정하여 CesiumJS 내장 일조 엔진 기반 그림자 시뮬레이션을 실행하며, 타임라인 슬라이더로 시간에 따른 그림자 변화를 실시간 확인한다.

**UC-07. 건축 규정 검토 및 결과 관리**
사용자는 배치된 건물에 대해 건폐율, 이격거리, 높이제한 규정 검토를 실행한다. 서비스 서버는 Shapely 기반 기하 연산으로 각 항목의 적합/부적합을 판정한다. 검토 결과는 항목별 현재값/기준값/상태로 표시되며, 위반 항목은 3D 뷰어에서 색상으로 시각적 강조된다. 사용자는 용도지역별 기준값을 직접 설정할 수도 있다. 최종 결과는 프로젝트에 저장하거나 PDF/이미지로 내보낼 수 있다.

#### ■ 관리자용 유스케이스 다이어그램

(※ 그림 2.3 관리자용 유스케이스 다이어그램 — PPT 또는 UML 툴로 작성 후 캡처하여 삽입)

관리자용 유스케이스는 아래와 같이 5개 주요 기능 그룹으로 구성된다.

**UC-A01. 관리자 인증**
관리자는 관리자 전용 계정으로 로그인하여 관리자 대시보드에 접근한다. 일반 사용자 계정과 권한이 분리되며, JWT 토큰에 관리자 역할(role: admin)이 포함된다.

**UC-A02. 사용자 관리**
관리자는 시스템에 등록된 전체 사용자 목록을 조회하고, 사용자 계정의 상태를 변경(활성/비활성/삭제)한다. 검색 필터와 페이지네이션을 지원한다.

**UC-A03. 프로젝트·데이터 관리**
관리자는 모든 사용자의 프로젝트를 조회·삭제하고, 업로드된 DXF 파일의 전체 목록을 관리한다. 외부 데이터 소스(지적도, 지형)의 연동 상태를 확인하며, 용도지역별 건축 규정 기준값의 기본 설정을 관리한다.

**UC-A04. 시스템 관리**
관리자는 서버 실행 로그 및 오류 로그를 조회하여 시스템 동작 상태를 확인한다. 파일 용량 제한, 동시 접속자 수 등 서비스 전반의 설정을 관리한다.

**UC-A05. 모니터링**
관리자는 CPU, 메모리, 네트워크 사용량 등 서버 리소스를 실시간으로 모니터링하고, 일별 접속자 수, 프로젝트 생성 수 등 서비스 사용 통계를 확인한다.

## 7. API 설계

### 7.1 서버/클라이언트 구조

본 시스템은 프론트엔드(Next.js)와 백엔드(FastAPI)를 분리한 클라이언트-서버 아키텍처를 채택한다. 프론트엔드는 Next.js App Router 기반의 SPA(Single Page Application)로 동작하며, CesiumJS 3D 뷰어를 통해 사용자 인터랙션을 처리한다. 백엔드는 FastAPI 기반 REST API 서버로, DXF 파일 처리, 3D 모델 생성, 건축 규정 검증 등의 비즈니스 로직을 담당한다.

주요 구조 요소:

- Data API Layer: FastAPI 라우터를 통한 RESTful API 제공. 클라이언트의 모든 데이터 요청을 처리하며, Pydantic 모델로 요청/응답 데이터를 검증한다.
- File Processing Module: DXF 파일 파싱(ezdxf), 좌표 변환(pyproj), 3D GLB 모델 생성(trimesh/pygltflib) 등의 파일 처리 파이프라인을 제공한다.
- Validation Module: Shapely 기반 기하 연산으로 건폐율, 이격거리, 높이제한 등 건축 규정을 검토한다. 용도지역별 기준값을 내장하여 자동 검증한다.
- Static File Server: 생성된 GLB 3D 모델 파일을 정적 파일로 서빙하여 CesiumJS에서 직접 로드할 수 있도록 한다.
- CORS Middleware: 프론트엔드-백엔드 간 크로스 오리진 통신을 허용한다.
- Database Layer (설계 예정): PostgreSQL + PostGIS를 사용하여 프로젝트, 건물, 사용자 데이터를 영구 저장한다. SQLAlchemy ORM과 GeoAlchemy2를 통해 공간 데이터를 처리한다.

(※ 구조도는 PPT로 작성 후 캡처하여 삽입)

## 8. 데이터 설계

### 8.1 ERD

(※ ERD는 ERD 툴(dbdiagram.io, MySQL Workbench 등) 또는 PPT로 작성 후 캡처하여 삽입)

본 시스템의 데이터베이스는 PostgreSQL 16 + PostGIS를 사용하며, 아래와 같은 테이블 구조를 가진다. DB명: `building_cesium_db`

ERD 요약:
- users (1) ─── (N) projects : 사용자는 여러 프로젝트를 생성할 수 있다
- projects (1) ─── (N) buildings : 프로젝트에는 여러 건물이 배치될 수 있다
- projects (1) ─── (N) dxf_files : 프로젝트에 여러 DXF 파일을 업로드할 수 있다
- dxf_files (1) ─── (N) buildings : DXF 파일에서 여러 건물 풋프린트를 추출할 수 있다
- projects (1) ─── (N) validation_results : 프로젝트에서 여러 번 규정 검토를 실행할 수 있다
- buildings (1) ─── (N) validation_results : 검토 결과는 특정 건물에 대해 생성된다

## 의논 필요 항목

### 팀 회의에서 결정해야 할 사항

**2.1 서비스 구성 및 개요**
- 서비스 시나리오 구성도를 그려야 하는데, AI학습서버와 AI서비스 서버를 분리해서 그려야 함 (양식 요구사항)
- AI 모델이 DXF 외곽선 추출에 사용되는 건지, 규정 검토 예측에 사용되는 건지 용도 확정 필요

**2.2 데이터분석의 요구사항**
- AI학습용 데이터가 뭔지 확정 필요: DXF 도면 데이터? 건축 규정 데이터? 건물 배치 데이터?
- 데이터 수집 출처, 양, 포맷을 정의해야 함

**2.5~2.6 요구기능 정의**
- 유스케이스 다이어그램을 먼저 확정해야 요구기능 표를 채울 수 있음
- 사용자용/관리자용 유스케이스를 누가 그리는지 확인 필요 (상현? 호민?)

**8.1 ERD / 8.2 테이블명세서**
- DB 스키마를 설계해야 하는데, 아직 DB가 미연동 상태
- 최소 필요 테이블: users, projects, buildings, dxf_files, validation_results, (ai_models?)
- ERD 그리기 전에 팀원들과 어떤 데이터를 저장할지 합의 필요
