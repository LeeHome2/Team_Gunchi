### 1. CesiumViewer.tsx 컴포넌트 분리

**현상**: CesiumViewer.tsx가 700줄 이상의 단일 파일로, 지적도 로직·모델 드래그·블록 선택·OSM Buildings·휴먼스케일 모델·경계 체크·상태 저장/복원 등이 전부 한 컴포넌트에 있음

**문제**: 기능 추가 시 충돌 위험 증가, 코드 탐색 어려움, 테스트 불가

**개선 방안**: 커스텀 훅으로 관심사 분리
```
frontend/hooks/
├── useCesiumInit.ts       # 뷰어 초기화, Cesium 스크립트 로딩, 기본 설정

├── useModelDrag.ts        # 3D 모델 드래그/회전 마우스 인터랙션
├── useBoundaryCheck.ts    # 모델 바운더리 체크, 색상 피드백
├── useBlockSelection.ts   # 블록 선택/해제 모드
└── useViewportState.ts    # localStorage 상태 저장/복원
```

**예상 효과**: CesiumViewer.tsx는 훅 조합만 담당하는 150줄 이내의 얇은 컴포넌트가 됨

### 2. main.py 규정 검토 로직 → 서비스 함수 호출로 변경

**현상**: `POST /api/validate-placement` 엔드포인트(166~240줄)에 Shapely 기하 연산이 직접 구현되어 있음

**문제**: `services/validation.py`에 동일 로직이 더 완성도 높게 구현되어 있음 (용도지역 지원, 대지 이탈 검사 포함). 코드 중복으로 수정 시 양쪽 모두 변경해야 함

**개선 방안**:
```python
# main.py — 변경 전
@app.post("/api/validate-placement")
async def validate_placement(request: ValidationRequest):
    from shapely.geometry import Polygon
    site_polygon = Polygon(request.site_footprint)
    building_polygon = Polygon(request.building_footprint)
    # ... 70줄의 직접 계산 로직

# main.py — 변경 후
@app.post("/api/validate-placement")
async def validate_placement(request: ValidationRequest):
    from services.validation import validate_placement as do_validate, ValidationConfig
    config = ValidationConfig(
        coverage_limit=request.coverage_limit or 60.0,
        setback_required=request.setback_required or 1.5,
        height_limit=request.height_limit or 12.0,
    )
    result = do_validate(
        request.site_footprint,
        request.building_footprint,
        request.building_height,
        config
    )
    return result
```

## 🟡 권장 개선 (코드 정리/일관성)

### 4. isPointInPolygon 함수 중복 제거

**현상**: CesiumViewer.tsx에 ray-casting 알고리즘이 2곳에 중복 구현
- 130줄: `checkInPolygon` (useEffect 내부 로컬 함수)
- 450줄: `isPointInPolygon` (useCallback 훅)

**개선 방안**: `lib/geometry.ts`로 추출하여 단일 함수로 통합
```typescript
// frontend/lib/geometry.ts
export function isPointInPolygon(point: [number, number], polygon: number[][]): boolean {
  const [x, y] = point
  let inside = false
  for (let i = 0, j = polygon.length - 1; i < polygon.length; j = i++) {
    const [xi, yi] = polygon[i]
    const [xj, yj] = polygon[j]
    if (((yi > y) !== (yj > y)) && (x < (xj - xi) * (y - yi) / (yj - yi) + xi)) {
      inside = !inside
    }
  }
  return inside
}
```

### 6. 좌표 변환 로직 중복 통합

**현상**: 경위도 ↔ 미터 변환이 3곳에서 각각 구현됨

| 위치 | 구현 | 방식 |
|------|------|------|
| CesiumViewer.tsx (148줄) | `111320 * Math.cos(latRad)` | 직접 계산 |
| gltf_exporter.py (194줄) | `footprint_to_cesium_coordinates()` | `1.0 / 111000.0` |
| coordinate_transform.py (124줄) | `meters_to_degrees()` | `EARTH_RADIUS` 기반 |

**문제**: 미세한 상수 차이 (111320 vs 111000), 유지보수 시 일관성 깨질 위험

**개선 방안**:
- 프론트엔드: `lib/coordinates.ts`에 통합 (이미 파일 존재, 활용도 높이기)
- 백엔드: `coordinate_transform.py`의 함수만 호출하도록 `gltf_exporter.py` 수정
- 상수 통일: `EARTH_RADIUS = 6378137.0`, `METERS_PER_DEGREE_LAT = π * R / 180`

### 7. Cesium 타입 안전성 개선

**현상**: Cesium을 `declare global { Window: { Cesium: any } }`로 처리하여 모든 Cesium API가 `any` 타입

**문제**: IDE 자동완성 불가, 컴파일 타임 오류 검출 불가, 오타로 인한 런타임 에러 위험

**개선 방안** (점진적):
```typescript
// 1단계: 자주 쓰는 Cesium 타입을 로컬에서 정의
interface CesiumViewer {
  camera: CesiumCamera
  entities: CesiumEntityCollection
  scene: CesiumScene
  clock: CesiumClock
  destroy(): void
}

// 2단계: cesium 패키지의 내장 타입 활용
import type { Viewer, Cartesian3, Entity } from 'cesium'
```

> ※ 현재 Cesium을 `<script>` 태그로 전역 로드하는 구조 때문에 바로 import 타입을 쓰기 어려움. 1단계(로컬 타입 정의)부터 시작 권장.
